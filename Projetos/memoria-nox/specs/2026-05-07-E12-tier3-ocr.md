# E12 — Tier 3 OCR pipeline

> Backlog finito de ~728 PDFs scaneados (~25k pages estimadas) que markitdown rejeita silenciosamente (retorna ~2 chars). Batch one-shot, não streaming. Custo é o gating concern. Spec define detection, engine choice trade-off, pipeline, idempotência, e gate criteria.

**TL;DR de board:** decisão é Tesseract local vs cloud OCR. Volume é suficientemente pequeno (~25k pages) pra cloud custar ≤ US$ 40 one-shot mas suficientemente grande pra Tesseract local levar 24-36h CPU. Recomendação: **Cloud (Google Document AI), one-shot, ~US$ 30-40, batch noturno**. Justificativa em §3.

---

## 1. Problema

`E02 Tier 2 PDFs` cobriu 79% (3.541/4.495) do corpus PDF via markitdown. **Gap residual ~728 PDFs**:

| Bucket | Volume | Origem |
|---|---|---|
| PPR | 372 | Pasta `~/Documents/PPR/` (relatórios scaneados, recibos) |
| PESSOAL | 250 | Pasta `~/Documents/PESSOAL/` (docs governo, contratos antigos) |
| Size-rejected | ~106 | A6 sweep rejeitou >50MB (multi-page scans) |
| **Total** | **~728** | — |

**Sintoma observado** (memória `feedback_scanned_pdf_heuristic`): markitdown sem OCR retorna conteúdo `<100 chars` (tipicamente 2-50) em PDFs imagem. A6 atual já flag-descarta esses arquivos. Não há ingestão alguma — chunks=0, vetores=0.

**Estimativa de pages:** assumindo média 30-40 pages/doc (skewed por contracts/relatórios longos) → ~22-29k pages totais. Para cálculo ancorado, usar **25.000 pages** como ponto de design.

**Não-problema:**
- Não é streaming live (não chega PDF novo todo dia)
- Não é OCR de imagens isoladas embutidas em chunks markdown já ingeridos
- Não é re-processamento de PDFs já absorvidos via texto-nativo

## 2. Detection layer

Trigger pra rotear pra OCR pipeline:

### 2.1 Heurística primária — pós-markitdown char count

```typescript
// src/lib/ocr-detector.ts
const SCANNED_PDF_CHAR_THRESHOLD = 100;

export function isScannedPdf(markitdownOutput: string, fileSize: number): boolean {
  if (!markitdownOutput) return true;
  const stripped = markitdownOutput.replace(/\s+/g, "").trim();
  if (stripped.length < SCANNED_PDF_CHAR_THRESHOLD) return true;
  // sanity: PDF >5MB com <0.001 chars/byte ratio é altamente provável scan
  if (fileSize > 5_000_000 && stripped.length / fileSize < 0.001) return true;
  return false;
}
```

### 2.2 Hint secundário — extension/path pattern

Pasta `PPR/` e `PESSOAL/` já tem prior alto. Permitir override manual via `--force-ocr` no CLI batch.

### 2.3 Pre-flight — pdftotext probe (zero custo)

Antes de invocar markitdown completo, rodar `pdftotext -l 1 file.pdf -` (poppler-utils, CLI). Se retorna `<50 chars` no primeiro page → flag `likely_scan=true`, pula markitdown, vai direto pra fila OCR. Economiza markitdown call em ~728 docs.

## 3. Engine choice — trade-off

### 3.1 Opção A — Tesseract local (CPU)

| Dimensão | Valor |
|---|---|
| Capex | ~US$ 0 (pacote `tesseract-ocr` + `tesseract-ocr-por` lang pack já no Debian) |
| Opex marginal | ~0 (CPU VPS Hostinger já provisionado) |
| Throughput | 3-5 sec/page CPU single-thread; 4 cores paralelos = ~1.2 sec/page efetivo |
| Total batch 25k pages | **~8-10h** CPU @ 4 cores; **~24-30h** single-thread |
| Qualidade PT-BR | Média. Lang pack `por` razoável em texto limpo, **degrada em tabelas, layouts complexos, scans baixa-resolução** |
| Tabela/layout reconstruction | Fraco (linhas viram fluxo desordenado) |
| Setup | `apt install tesseract-ocr tesseract-ocr-por poppler-utils` + wrapper pdf→imagens (`pdftoppm`) → tesseract por page |
| Operacional | Bloqueia 1 core VPS por horas; competing com nox-mem-api/watcher (já carga ativa) |

**Quando Tesseract ganha:** offline-mandatory, custo zero hard-constraint, qualidade média aceitável pro caso de uso.

### 3.2 Opção B — Google Document AI

| Dimensão | Valor |
|---|---|
| Pricing (2026-05) | US$ 1.50 / 1.000 pages (OCR processor) |
| Pricing 25k pages | **~US$ 37.50** one-shot |
| Throughput | ~30k pages/h (paralelizável até quota) |
| Total batch 25k pages | **~30-60min** wall-clock incluindo upload |
| Qualidade PT-BR | Alta. Lang detection automático, PT-BR first-class |
| Tabela/layout | Excelente — `OCR_PROCESSOR` retorna structured layout (paragraphs, tables, form fields) |
| Setup | gcloud SDK + service account JSON + Document AI processor instance + bucket GCS para upload temp |
| Operacional | Não bloqueia VPS; requer egress (uploads ~5-15GB de PDFs) |

### 3.3 Opção C — AWS Textract

| Dimensão | Valor |
|---|---|
| Pricing (2026-05) | US$ 1.50 / 1k pages (DetectDocumentText básico); US$ 15.00 / 1k pages (AnalyzeDocument com forms+tables) |
| Pricing 25k pages básico | ~US$ 37.50 |
| Throughput | Similar ao Google |
| Qualidade PT-BR | Boa, mas **historicamente atrás de Google em PT-BR** |
| Tabela/layout | Ótimo no tier `AnalyzeDocument`, mas custo 10× |

### 3.4 Opção D — Azure Computer Vision (Read API)

| Dimensão | Valor |
|---|---|
| Pricing (2026-05) | US$ 1.00 / 1k pages (S1 tier, 1k+ transactions/month) |
| Pricing 25k pages | **~US$ 25** |
| Qualidade PT-BR | Média-alta |
| Setup | Azure subscription + Cognitive Services key |

---

### 3.5 Recomendação

**Google Document AI** com fallback Tesseract local pra docs onde cloud falhar (timeout/rate-limit/conteúdo sensível flagged):

1. **Custo absoluto baixo** (~US$ 37.50 total) — abaixo do threshold de 1 hora de CFO time
2. **Qualidade PT-BR superior** importa: corpus inclui contratos, lições, decisões — degradação 10-15% recall em search downstream se OCR for ruim
3. **Wall-clock <1h** vs 8-30h Tesseract — libera VPS pra workload normal
4. **Structured layout output** preserva tabelas — relevante pra docs PPR (relatórios estruturados)
5. **One-shot batch** — não há ROI em otimizar custo recorrente porque não há recorrência

**LP-style framing:** US$ 37.50 é trivial; a verdadeira pergunta é qualidade + tempo. Tesseract economiza US$ 37.50 mas adiciona 24h+ wall-clock de incerteza (qualidade + ops) sobre um asset (memória corporativa) que vale ordens de grandeza mais. Skip the false economy.

**Edge case — docs sensíveis:** PESSOAL/ pode conter docs identidade/CPF/dados bancários. **Trigger explícito de Toto** antes de upload pra cloud. Default: Tesseract local pra `~/Documents/PESSOAL/`, cloud pra resto. Override `--allow-cloud-pessoal` se Toto autorizar caso-a-caso.

## 4. PT-BR considerations

| Aspecto | Tesseract `por` | Google Doc AI |
|---|---|---|
| Acentuação (ã, ê, ç) | Razoável; falha em scans <300dpi | Robusto |
| Layout 2-col (jurídico) | Quebra em fluxo único | Detecta colunas |
| Tabelas (R$ valores) | Vira string desordenada | Structured `Table` object |
| Carimbos/assinaturas | Vira ruído inserido inline | Detecta como `image_only` regions, ignorável |
| Manuscrito | Falha total | Limited (não confiar) |

**Implicação ingest:** OCR markdown output deve preservar `\n\n` entre logical sections. Para tables, converter Document AI Table → markdown table na conversão pra `.md`.

## 5. Pipeline

```
PDF source
    │
    ├─► pdftotext -l 1 probe (zero-cost char count)
    │       │
    │       ├─ chars ≥100 ─► markitdown path (Tier 2 existing) ─► routeIngest
    │       │
    │       └─ chars <100 ─► OCR queue
    │                           │
    │                           ├─ document_ai_process(file)
    │                           │       │
    │                           │       └─► markdown render (text + tables)
    │                           │               │
    │                           │               └─► hash dedup check ────────┐
    │                           │                                              │
    │                           └─ on error/timeout ─► tesseract_fallback     │
    │                                                       │                  │
    │                                                       └─► markdown ─────┤
    │                                                                          │
    └─────────────────────────────────────────────────────────────────────────►│
                                                                                │
                                                                       routeIngest()
                                                                                │
                                              section auto = NULL (legacy markdown)
                                              ocr_status = 'cloud' | 'tesseract' | 'hybrid'
                                              source_path canonical preserved
                                                                                │
                                                                       chunks + FTS5 + vec
```

### 5.1 routeIngest extension

Adicionar handler em `src/lib/ingest-router.ts`:

```typescript
// Pseudocódigo
if (file.endsWith(".pdf") && isScannedPdf(probe, size)) {
  return ingestOcrFile(file, opts); // novo handler
}
if (file.endsWith(".pdf")) {
  return ingestMarkitdown(file, opts); // path existente (Tier 2)
}
```

`ingestOcrFile()`:
1. Renderiza PDF → markdown (cloud ou tesseract)
2. Salva `.md` em `cache/ocr/<sha256>.md` (idempotência + audit trail)
3. Chama `ingestMarkdown(cachedPath, { ...opts, originalSourcePath: file })`
4. Anota `ocr_status` no chunk metadata

### 5.2 Schema v15 (additive)

```sql
ALTER TABLE chunks ADD COLUMN ocr_status TEXT;
-- valores: NULL (não-OCR), 'cloud' (Google Doc AI), 'tesseract' (local fallback),
-- 'hybrid' (cloud com partial fallback), 'failed' (registrado mas zero text)

CREATE TABLE IF NOT EXISTS ocr_jobs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  source_path TEXT NOT NULL,
  source_sha256 TEXT NOT NULL UNIQUE,
  pages_count INTEGER,
  engine TEXT NOT NULL,                -- 'google-doc-ai' | 'tesseract' | 'azure' | 'aws'
  status TEXT NOT NULL,                -- 'queued' | 'running' | 'success' | 'failed'
  cost_usd REAL,
  duration_ms INTEGER,
  output_md_path TEXT,                 -- cache/ocr/<sha>.md
  chunks_ingested INTEGER DEFAULT 0,
  error_msg TEXT,
  created_at TEXT DEFAULT (datetime('now')),
  completed_at TEXT
);

CREATE INDEX idx_ocr_jobs_status ON ocr_jobs(status);
CREATE INDEX idx_ocr_jobs_engine ON ocr_jobs(engine);
```

`source_sha256 UNIQUE` é a guarda de idempotência — re-run do batch não re-processa nem re-cobra.

## 6. Idempotência + dedup

### 6.1 Dedup vs versão sem-OCR

Algum subset dos 728 PDFs pode ter sido **parcialmente** ingerido com markitdown garbage (~2 chars stripped → talvez algum frontmatter PDF metadata vazou pra chunks). Antes de ingerir versão OCR'd:

```sql
-- Soft-delete chunks de mesmo source_path com texto curto (<100 chars)
UPDATE chunks
SET deleted_at = datetime('now')
WHERE source_path = ?
  AND length(text) < 100
  AND ocr_status IS NULL;
```

Trigger `trg_chunks_delete_cascade` (já existente) limpa vec_chunks/FTS automaticamente.

### 6.2 Idempotência de batch

Pre-flight de cada item:
```sql
SELECT id, status, output_md_path FROM ocr_jobs WHERE source_sha256 = ?;
```
- `success` → skip (já feito)
- `running` (orphan, run abortou) → reset → re-process
- `failed` → re-tentar se age > 1d (transient errors expirados)
- não existe → process novo

### 6.3 Idempotência por engine

Se cloud falhou em N docs e tesseract os processou, re-run `--engine google` não re-cobra (tem `success` row). Para forçar re-OCR: `--force-reprocess` flag explícito.

## 7. Cost model explícito

| Cenário | Engine | 25k pages | Wall-clock | Risco qualidade |
|---|---|---|---|---|
| **A — Cloud only** | Google Doc AI | **~US$ 37.50** | ~45min | Baixo |
| **B — Cloud + tesseract fallback** | Google primary, tesseract on error | ~US$ 35-37 | ~1h | Baixo |
| **C — Tesseract only** | Local CPU | **~US$ 0** | 8-30h | Médio (PT-BR scans <300dpi degrada) |
| **D — Azure Read** | Azure CV | **~US$ 25** | ~1h | Baixo-médio |
| **E — Hybrid pré-classificado** | Tesseract na PESSOAL/, cloud no resto | ~US$ 25-28 | ~6-10h (gargalo PESSOAL local) | Privacy-aware |

**Cenário escolhido:** **B (Cloud + tesseract fallback)** com **subvariante E** ativada para PESSOAL/ se Toto não autorizar upload.

**Custo upper bound budget:** US$ 50 (margem +33% sobre estimate). Se ultrapassar, abort + investigar (page count > estimado, processador errado).

## 8. Gate criteria — quando declarar E12 done

| # | Critério | Threshold | Como medir |
|---|---|---|---|
| 1 | % PDFs gap absorvidos | ≥ 95% (≥ 692/728) | `SELECT COUNT(*) FROM ocr_jobs WHERE status='success'` |
| 2 | % páginas OCR'd com texto extraído | ≥ 90% | `SUM(chunks_ingested > 0) / COUNT(*)` |
| 3 | Chunks net adicionados | ≥ 8.000 (assumindo ~12 chunks/doc média) | DB diff pré/pós |
| 4 | Vectorize coverage pós-batch | == 100% | `/api/health.vectorCoverage` |
| 5 | Recall em queries scan-dependent | nDCG@10 ≥ 0.40 em sub-corpus | curar 5 golden queries que dependem de docs PPR/PESSOAL |
| 6 | Cost real | ≤ US$ 50 | `SUM(cost_usd) FROM ocr_jobs` |
| 7 | Zero corruption schema | passa invariants | `bash scripts/check-schema-invariants.sh` |
| 8 | Backup pre-batch existe | required | `withOpAudit()` snapshot em `/var/backups/nox-mem/pre-op/` |

**Pass:** todos critérios → flip ROADMAP linha E12 ✅ DONE, anotar em `docs/EVOLUTION.md` (schema v15 entry), arquivar spec.

**Partial pass:** se #5 (recall) falha mas #1-4 OK → não-bloqueador; criar follow-up E12b (rerank/expansion tuning), não re-rodar OCR.

## 10. Não-objetivos (out of scope)

| Item | Por quê não |
|---|---|
| OCR streaming live (incoming PDFs) | Volume não justifica daemon; basta cron mensal `nox-mem ocr-batch ~/Documents --since-last-run` se aparecer drift |
| OCR de imagens isoladas em chunks markdown já ingeridos (ex: screenshot embedded) | Esses já viraram chunks via markitdown — re-extração teria de re-parse o markdown, escopo diferente |
| Re-processamento de PDFs que já viraram texto via Tier 2 | Markitdown sucesso é status quo; nao quebrar |
| Manuscritos / handwriting | Fora do scope. Se PESSOAL/ tem cartas manuscritas, marcar `ocr_status='failed'` e seguir |
| Audio/video transcription (gated em Tier 4 separado) | Pipeline OCR é text-from-image; transcrição é speech-to-text, separar em E14 ou similar |
| OCR de outros formatos (TIFF isolado, PNG isolado fora de PDF) | Volume não-relevante; PDFs cobrem o gap real |
| Cloud OCR multi-region failover | One-shot, não recorrente; single region GCP é suficiente |
| Real-time cost monitoring dashboard | F10 dashboard cobre se reativado; pra batch one-shot, log + reconcile pós-fato basta |
| Auto-tagging via LLM dos docs OCR'd (KG injection no batch) | Cron noturno `kg-build incremental` cobre dia seguinte, sem custo marginal de spec |

## 11. Risk register

| # | Risco | Severity | Mitigação |
|---|---|---|---|
| 1 | Custo real >> estimate (page count subestimado) | Médio | `--dry-run` + `--budget-cap` flag aborta se > US$ 50 |
| 2 | Doc AI falha em N% docs → fallback tesseract massivo → 24h+ wall-clock inesperado | Médio | Stats per-engine no `ocr_jobs`; abort batch se fallback rate >10% |
| 3 | PDF PESSOAL/ vaza dado sensível pra cloud sem autorização | **Alto** | Default-deny: PESSOAL/ usa tesseract; flag explícita `--allow-cloud-pessoal` |
| 4 | OCR markdown garbage poluí FTS e degrada recall global | Médio | Canary 5 docs antes de full run + gate recall #5 valida |
| 5 | Schema v15 migration corrompe DB | Alto | `withOpAudit()` snapshot pré-migration (regra crítica #6) + `safeRestore()` recovery path |
| 6 | Tesseract `por` lang pack não instalado na VPS | Baixo | Pre-flight check; `apt install tesseract-ocr-por` se faltando |
| 7 | Batch trava 80% e abort — re-run cobra cloud de novo? | Médio | Idempotência via `ocr_jobs.source_sha256 UNIQUE` (§6.2) |
| 8 | Disk full (cache/ocr + chunks + vec3072d) | Médio | Pre-flight: `df -h` ≥10GB livre; abort se não |
| 9 | Documentos identidade/CPF triggerar GCP DLP auto-redaction sem aviso | Baixo-médio | Toto autoriza ou usa tesseract; checar se Doc AI processor tem DLP enabled (default: não) |
| 10 | OCR introduz texto incorreto e KG extraction injeta entities falsas | Médio | KG extraction é incremental + reviewable; spot-check 10 entities pós-batch |
