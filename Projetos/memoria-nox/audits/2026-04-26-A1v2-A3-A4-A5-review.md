## Lições da sessão

1. **Audit duplo cobre código standalone, não estado de produção.** A1 v2 audit em 04-25 achou 27 findings + fixou 5 CRIT/HIGH no código; mas o estado real `/var/backups/nox-mem/pre-op/` em prod estava `755` por drift histórico, e snapshots manuais antigos `644`. Próximas sessões: adicionar verificação de **prod state** ao audit (stat + getfacl no dir alvo).
2. **chmod silencioso (catch {} vazio) em código de segurança = bug ativo invisível.** O original `try { chmodSync(...); } catch {}` mascarava qualquer falha de chmod cross-FS ou perm-denied. Agora fail-closed → primeiro snapshot que falhar chmod aborta op em vez de gerar leak window.
3. **Path validation precisa cobrir TODOS argumentos de funções de write.** A1 v2 validava `NOX_PRE_OP_SNAPSHOT_DIR` no withOpAudit path, MAS safeRestore (recovery path, menos exercitado) aceitava qualquer `snapshotPath`. Asymmetric trust = bug.
4. **Canary com false-positive risk dispara mais alerts que valor real.** `set -u` sozinho permitia empty result interpretado como 0 → spam Discord. Hoje toda canary nova precisa: `-uo pipefail`, sentinel pra empty (não 0), dedup window.
5. **Reaper coverage = topology gap.** B1 (sessão anterior) já mostrou que `reapZombies()` em só 1 startup path deixava CLI invocations descobertas; `reindex` em loop noturno acumulava zumbis. Quando adicionar features que tracking state cross-process, perguntar: "qual processo é responsável pelo cleanup?" ANTES de codar.
