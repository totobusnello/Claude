## 3. Cryptographic Design

### 3.1 Algorithm stack (locked)

| Layer | Choice | Justification |
|---|---|---|
| Cipher | AES-256-GCM | AEAD native to Node `crypto.createCipheriv`; tamper-detecting via 16-byte auth tag; FIPS-validatable. |
| KDF | scrypt | Memory-hard; native (`crypto.scryptSync`); no dep. Tier 1 N=2^14 (~150ms laptop) â€” Tier 2 will raise per kickoff Â§1. |
| Salt | random 16 bytes per export | Standard size; stored base64 in manifest header. |
| IV / nonce | random 12 bytes per export | GCM standard length; **never** reused. Generated by `crypto.randomBytes`. |
| Auth tag | 16 bytes (default GCM length) | Bundle rejection on any one-byte ciphertext flip â€” empirically tested in case 2. |
| AAD | `sha256(canonical(manifest_header))` | Locks counts, salt, IV, timestamp, KDF params into the auth tag. Tamper on any field â†’ AAD diverges â†’ import fails before decrypt is even attempted. |

### 3.2 Bundle envelope

```jsonc
{
  "version": 1,
  "created_at": "<iso8601>",   // AAD-bound
  "chunks_count": N,           // AAD-bound
  "entities_count": N,         // AAD-bound
  "relations_count": N,        // AAD-bound
  "encrypted": true,
  "cipher": "aes-256-gcm",
  "key_derivation": "scrypt",
  "kdf_params": { "N": 16384, "r": 8, "p": 1, "keylen": 32 },  // AAD-bound
  "salt": "<b64>",             // AAD-bound
  "iv": "<b64>",               // AAD-bound
  "authTag": "<b64>",
  "aad": "<b64>",              // sha256 of the canonical header above
  "ciphertext": "<b64>"        // encrypted JSON: {chunks, entities, relations}
}
```

**Canonicalization** (`canonicalJson`): recursive sorted-keys with no whitespace. This is mandatory for AAD stability â€” two engines must produce byte-identical header buffers from the same logical content. Tested implicitly in case 1 (cross-instance roundtrip).

### 3.3 Defense layers

1. **AAD recompute before decrypt**: import recomputes `sha256(header)` locally and `timingSafeEqual`s vs the claimed AAD. Any modification to a header field (e.g. `chunks_count++`) is caught here with `AAD_MISMATCH` â€” without exposing the cipher to attacker-chosen AAD (case 4).
2. **GCM auth tag**: any flipped byte inside `ciphertext` is detected by the tag at `decipher.final()` (case 2). The error is wrapped as `TAMPERED_BUNDLE`.
3. **Count cross-check**: after decrypt, payload lengths are matched against header counts. A theoretical AAD-equivalent attack that produced a valid decrypt with wrong counts would still fail here (`TAMPERED_BUNDLE`). Defense in depth.
4. **Length validation**: salt (16), IV (12), auth tag (16) lengths are validated before key derivation begins. Malformed bundles fail with `BAD_BUNDLE_FORMAT`.
5. **Min passphrase length**: 8 chars at export (`WEAK_PASSPHRASE`). Tier 2 will raise to 16 + entropy check.
6. **Wrong passphrase indistinguishable from tamper** by design: both surface `TAMPERED_BUNDLE`. An attacker cannot distinguish "bad password" from "I successfully altered the bundle" â€” eliminates an oracle.

### 3.4 Passphrase handling (no leakage)

- CLI rejects `--passphrase`, `--passphrase=<val>`, `-p`, `-p=<val>` at argv parse time (case 10). Reason: `ps aux` leak.
- Only `--passphrase-env <ENV_VAR_NAME>` is accepted. The CLI resolves the value at runtime from `process.env[name]`.
- Env var **name** is regex-validated (`^[A-Z_][A-Z0-9_]*$`) before lookup.
- Empty/missing env var â†’ fail closed with exit code 2.
- Memory wipe (`key.fill(0)`) after derive + use. Best-effort â€” Node's GC may keep copies; documented as Tier 2 followup if hardening required.

## 5. Schema Awareness

The Tier 1 implementation snapshots all rows of `chunks`, `kg_entities`, `kg_relations` via `SELECT *`. Columns are introspected at import time via `pragma table_info(...)` and matched by name â€” **drift-tolerant**: source columns missing in target are silently dropped (logged in Tier 2 audit); target columns missing in source are left as their schema defaults.

The full schema v.29 column list (per CLAUDE.md) is preserved verbatim:
- `chunks`: `id, content, content_hash, source, section, section_boost, pain, retention_days, created_at, ...`
- `kg_entities`: `id, canonical_name, type, frontmatter_json, ...`
- `kg_relations`: `id, source_entity_id, predicate, target_entity_id, confidence, ...`

**Out of scope for Tier 1 (deferred):**
- FTS5 + sqlite-vec rebuild after `replace` strategy (caller must `nox-mem reindex --no-mutate-section` post-import; Tier 2 will wire automatically).
- `ops_audit` append-only serialization (kickoff T6).
- KG `source_entity_id` / `target_entity_id` FK remapping in merge mode (kickoff T5). Tier 1 keeps source ids; merge with non-empty target may produce dangling refs â€” documented as Tier 1 limitation.

In **replace** mode the source ids are preserved (truncate first), so FK refs inside the bundle resolve naturally.
In **merge** mode the `id` column is **dropped** so SQLite auto-assigns fresh ids â€” this prevents spurious PK collisions but means inter-table FK refs would dangle. This is the documented Tier 1 trade-off and Tier 2 ships the ID-remapping pass.

## 7. Rollback Paths

| Failure mode | Rollback |
|---|---|
| Export crashes mid-write | Bundle is written via `writeFileSync` (atomic at POSIX level for small files); partial write leaves no DB state changed. No rollback needed. |
| Import crashes mid-insert | All mutations wrapped in `db.transaction(...)` â€” better-sqlite3 rolls back automatically on throw. |
| Import succeeds but data is wrong | `nox-mem restore <pre-op-snapshot>` from withOpAudit. **NOTE:** Tier 1 import is NOT yet wrapped in withOpAudit â€” that's a Tier 2 task. For Tier 1 users should snapshot manually before running `--strategy replace` on prod data. |
| Bundle file gets corrupted in transit | GCM auth tag catches single-byte corruption (case 2); checksums per file in Tier 2 catch multi-byte corruption with file-level error. |
| Forgot passphrase | **No recovery.** Documented in audit. Recommended: store passphrase in password manager + keep one unencrypted snapshot as DR (kickoff Â§16 risk). |
