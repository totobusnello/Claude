## Context — o que temos hoje

Auditoria rápida da VPS (2026-04-19) revelou base muito maior do que inicialmente suposto:

**Cada agente tem 10 arquivos de identidade** em `/root/.openclaw/workspace/agents/<name>/`:
`SOUL.md · IDENTITY.md · AGENTS.md · BOOTSTRAP.md · CHANNELS.md · TEAM.md · USER.md · HEARTBEAT.md · MEMORY.md · TOOLS.md`

**Discord já roteado:** 7 targets "discord" no `openclaw.json`; 1 `DISCORD_BOT_TOKEN` + 1 `DISCORD_WEBHOOK` — consistente com modelo "1 bot, personas via webhooks" (identidade visual por agente).

**Agentes e uso real (declarado pelo usuário):**
| Agente | Papel declarado | Uso real |
|---|---|---|
| Nox | Chief-of-staff, coordenador WhatsApp+Discord | Diário, intenso |
| Boris | LinkedIn + notícias | Diário |
| Forge | Desenvolvimento | Diário |
| Lex | Contratos | Ocasional |
| Atlas | Research + Valuation + Apresentações (SOUL muito rico — M&A, FII, DCF, LBO, cenários) | Pouco — gap de ativação, não de papel |
| Cipher | Segurança | Raro |

**Diagnóstico:** não há gap de identidade (SOULs existem e são ricos). Há gap de (a) **staleness/drift** desconhecido dos 10 arquivos, (b) **ativação proativa** — Nox não sugere dispatch quando deveria.

## Goals

1. Descobrir o estado real dos 10 arquivos × 6 agentes (audit)
2. Consolidar em estrutura canônica sem perder o que já funciona
3. Ativar os sub-usados (Atlas, Cipher, Lex) via Nox proativo + cron específico (Cipher)
4. Tornar A2A persistente e buscável (inbox table no nox-mem)
5. Medir ativação — saber depois se a intervenção funcionou

## Non-goals (explicit)

- Criar bot Discord novo — já existe
- Redefinir papel de qualquer agente — SOULs estão ricos, não refatorar
- Arquitetura mesh flat — mantém hub-and-spoke via Nox
- Bot separado por agente — personas via webhooks é mais enxuto

## Exit criteria (medidos)

- **Audit concluído** com tabela de 60 células (6 agentes × 10 arquivos) + recomendação por linha
- Estrutura canônica acordada; drift entre `CHANNELS.md` e `openclaw.json` = 0
- Nox sugere dispatch em ≥80% das conversas que match os patterns do TEAM.md (verificável em logs)
- Atlas dispatch rate ≥3×/semana em 30 dias pós-deploy (de 0-1 hoje)
- Cipher weekly audit rodando 4 domingos consecutivos sem falha
- `agent_inbox` com FTS funcional; busca "o que Forge pediu esta semana" retorna em <100ms
- Zero regressão no budget: $5/dia warn mantém sem disparar pela 1ª semana (telemetria)

## Depends on / blocks

- **Depende de:** Fase 1.7a (ontology grounding — entity type "agent" com schema rico)
- **Bloqueia:** Nada crítico. Path A continua pre-req pra Fase 2 independentemente.

## Risks + mitigations

| Risco | Mitigação |
|---|---|
| Audit revela caos maior que imaginado (staleness severa, drift grande) | Escopo: Fase 1.8a = só audit + doc; Fase 1.8b = consolidação. Decide depois do audit |
| Nox proativo vira chato ("quer passar pro Atlas?" a cada mensagem) | Confirm-by-default só no início; telemetria ativa; se >30% dos user replies = "não" → threshold aperta |
| Prompt injection cross-agent via inbox (Boris injeta instrução maliciosa no inbox do Forge) | Input validation em `dispatch_to`; body sanitizado; from_agent sempre trusted (não vem do Discord user) |
| Budget explode | Cap semanal por par (ex: Atlas↔Forge max 10 msgs/dia); tier Haiku forçado em A2A chatter |
| Consolidation perde conteúdo importante | Backup de todos os 60 arquivos pré-refactor; migration script reversível |

## Rollback

- Audit: read-only, sem rollback
- Consolidation: restore dos 60 arquivos de `agents-backup-pre-1.8/`
- Nox proativo: `UPDATE meta SET value='false' WHERE key='nox_proactive_dispatch'` (toggle sem deploy)
- Inbox: DROP TABLE + rollback schema v6 (mantém 1.6 intacto)
- Cipher cron: `crontab -e` remove linha
