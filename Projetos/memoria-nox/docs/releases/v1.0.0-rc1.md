## § 1 — What is nox-mem?

nox-mem is a production-grade, FOSS-first memory layer for LLM agents. It runs on your hardware, your storage, your provider — no vendor lock-in by design. The architecture is grounded in three pillars:

| Pillar | Promise |
|---|---|
| **Q — Quality** | Retrieval numbers first: hybrid BM25 + Gemini semantic + RRF, benchmarked against LongMemEval and LoCoMo |
| **A — Autonomy** | SQLite on your disk, swap the embedding provider, zero SaaS dependency |
| **P — Product** | CLI, HTTP API, MCP Server, and observability dashboards — ready to ship |

## § 2 — What's new in this release

### Q/A/P Architecture (pivot 2026-05-17)
Replaced the previous product-first framing with a three-pillar architecture. Quality gates now precede GTM. Phase 2 unlocked after D43: G5 V3 A8 returned +18.8% above the minimum threshold.

### G-series ablation gauntlet (G3 → G10d)
Ten validated ablation rounds on top of a production corpus (68,995 chunks, n=100):

| Round | Change | Delta nDCG@10 |
|---|---|---|
| G3 baseline | A0 vanilla hybrid | reference |
| G4 Wave A | Full boost stack A8 | +63.5% |
| G5 V3 | Production corpus A8 | **+78.8%** |
| G8 | `source_type_boost` | +2.66% (isolated) |
| G9 | Redundancy audit | Confirmed A5 +14.2% on prod corpus |
| G10 | Hard Mutex conditional | +0.79% nDCG / +2.65% MRR |
| G10b | Per-category mutex | single-hop +8.22%, open-domain +2.42% |
| G10c | Per-style mutex | natural-language +1.56% WIN |
| G10d | Conditional threshold=2 | multi-hop +1.58%, adversarial +3.04% recovered |

Full methodology: [`docs/Q4-COMPARISON-METHODOLOGY.md`](../Q4-COMPARISON-METHODOLOGY.md)

### Hard Mutex conditional — D51 (deployed prod)
`NOX_MUTEX_QUERY_ENTITY_THRESHOLD=2` — query-entity count gate prevents mutex over-application on keyword and adversarial queries where it previously caused regression. Deployed via PR #198 + #203.

### F10 Phase A + B — Observability dashboards (LIVE)
Two static HTML dashboards served from the HTTP API:

- `/observability/health.html` — prod health: chunk counts, vector coverage, KG stats, salience mode, ops_audit activity (PR #207)
- `/observability/evals.html` — eval timeseries: nDCG@10 + MRR line charts with gate annotations (PR #212)

### Pain-weighted salience (active mode)
Formula: `salience = recency × pain × importance × access_count_bonus`

Active via `NOX_SALIENCE_MODE=active`. Previously deployed in shadow mode (G7 confirmed formula is neutral on 500-chunk corpus; full benefit expected at 250k+ scale). Exposed via `/api/health.salience`.

### L4 plural normalisation — D52
Regex-first typed-link extraction (T0–T3) with 17 canonical entity types. Filesystem directories accept plural forms via `NOX_ENTITY_DIRS_PLURAL`. Extraction method logged on `kg_relations.extraction_method`. First cron fire: Sun 2026-05-25 23h UTC. (PR #209, #214, #225)

### G12 R3 — Layer-4 dedup carve-out
Entity chunks are now excluded from the cross-chunk deduplication path. Prevents false-positive dedup of high-signal compiled/frontmatter sections. (PR #206)

## § 4 — Technical changelog

### Schema
- `chunks.pain` REAL DEFAULT 0.2 — severity scale 0.1 (trivial) → 1.0 (prod outage) (Schema v9)
- `chunks.section` TEXT + `chunks.section_boost` REAL — entity section type: `compiled` (2.0×), `frontmatter` (1.5×), `timeline` (0.8×), legacy/NULL (1.0×) (Schema v10)
- `chunks.retention_days` — typed decay: `feedback`/`person` NULL (never-decay), `lesson` 180d, `decision`/`project` 365d, `team` 120d, `daily` 90d, `pending` 30d (Schema v8)
- `kg_relations.extraction_method` — L4 audit column for typed-link provenance
- `ops_audit` table — append-only operation audit with VACUUM INTO snapshot pre-op
- Trigger `trg_chunks_delete_cascade` — DELETE on chunks cascades to vec_chunks
- Trigger `trg_ops_audit_no_delete` + `trg_ops_audit_no_terminal_update` — CWE-693 enforcement

### Search
- **BM25 + Gemini semantic + RRF fusion** (k=60) — three-layer hybrid
- `section_boost` multiplicative per chunk (deployed G4 Wave A)
- `source_type_boost` — entity vs markdown source weighting (G8, isolated +14.2% on prod corpus)
- **Hard Mutex conditional** — suppresses section_boost on entity chunks when `query_entities < NOX_MUTEX_QUERY_ENTITY_THRESHOLD` (default 2); per-category and per-style variants validated G10b/G10c (PR #182, #198, #203)
- **G12 R3 entity dedup carve-out** — entity chunk type excluded from cross-chunk dedup (PR #206)
- FTS5 sanitizer: Unicode whitelist `/[^\p{L}\p{N}\s]/gu` (fixes zero-recall on NL queries with punctuation)

### Salience
- Formula: `recency_score × pain_score × importance_score × access_count_bonus`
- Active via `NOX_SALIENCE_MODE=active` (was `shadow` in all prior versions)
- Exposed via `/api/health.salience` — salience stats + mode
- G7 validation: formula is neutral on 500-chunk eval corpus; expected gains at 250k+ scale

### Knowledge Graph
- `kg_entities` + `kg_relations` with Gemini 2.5 Flash extraction (incremental nightly)
- L4 typed-link extraction: regex-first (T0–T3) with Gemini fallback (shadow-mode)
- 17 canonical entity types with plural filesystem normalisation (`NOX_ENTITY_DIRS_PLURAL`)
- `kg_build`, `kg-prune`, `cross-kg` CLI commands + `/api/kg`, `/api/kg/path`, `/api/cross-kg` endpoints

### API
- `/api/health` — full system health: chunk stats, vector coverage, KG stats, salience, ops_audit 24h counts
- `/api/search` — hybrid search with RRF, all boosts, mutex conditional
- `/api/answer` — RAG answer primitive (LIVE 2026-05-18, first answer ~1.6s latency)
- `/api/crystallize` + `/api/crystallize/validate` — memory crystallisation
- `/api/reflect` + `/api/procedures` — reflection and procedure retrieval
- `/observability/health.html` + `/observability/evals.html` — F10 Phase A + B dashboards

## § 5 — Breaking changes

This is the first public release. There are no breaking changes from a prior public version. For teams migrating from internal/dev builds:

| Area | Change | Action required |
|---|---|---|
| Schema | Fields `pain`, `section`, `section_boost`, `retention_days` added (v8-v10) | Auto-applied at first run via migration |
| Config | `NOX_SALIENCE_MODE=active` is the new default (was `shadow`) | Set `NOX_SALIENCE_MODE=shadow` to preserve prior behavior |
| Config | `NOX_MUTEX_QUERY_ENTITY_THRESHOLD=2` new conditional mutex | Set to `0` to disable mutex entirely |
| Config | `NOX_ENTITY_DIRS_PLURAL` opt-in (Tier-2) | Not required; enables plural dir forms for L4 |
| API | `/api/answer` added at :18802 | No conflict; port unchanged |
| CLI | Entry point is `dist/index.js` (not `cli.js`) | Update any direct invocations |
| API port | `:18802` (not 18800 — Chrome squats 18800) | Verify `NOX_API_PORT` in `.env` |

## § 6 — Known issues

1. **Pandoc strict mode + Unicode math** — 7 Unicode math characters outside `$...$` blocks cause `pandoc --strict` to reject the paper PDF. Fix: `--pdf-engine=xelatex`. Tracked in PR #234.

2. **L4 first cron fire pending** — `trg_extraction_method_backfill` and L4 regex-first path fire on the next cron cycle: Sun 2026-05-25 23h UTC. Watchpoint Mon 2026-05-26 morning. 21,518 relations currently have `extraction_method = NULL`.

3. **BibTeX URL verification pending** — 6 `[VERIFY-TUE-06-02]` entries in `paper/refs.bib` need manual URL check before arXiv submit.

4. **Multi-session / temporal queries** — LongMemEval multi-session and temporal sub-categories are weaker (identified Q2 2026-05-19). Temporal spike v2 (PR #181, shadow-mode) deployed but not yet active. Active promotion gated on 7-day shadow window.

5. **Benchmark gap vs EverMemBench** — nox-mem has not been evaluated on EverMemBench (EverMind-AI's published benchmark). Planned Lab Q3 2026.

## § 7 — Upgrade notes

### New installation
See [`docs/QUICKSTART.md`](../QUICKSTART.md) for setup. Minimum requirements: Node.js 20+, SQLite 3.45+, a Gemini API key.

### Upgrading from internal dev versions
1. Read [`docs/HANDOFF.md`](../HANDOFF.md) for current production state.
2. Schema migrations are auto-applied on first run — no manual SQL required.
3. Review `NOX_SALIENCE_MODE` — now defaults to `active` (was `shadow`).
4. Review `NOX_MUTEX_QUERY_ENTITY_THRESHOLD` — set to `0` if you want pre-G10 behavior.
5. Run `nox-mem reindex --dry-run` before any full reindex on a live corpus.

### API compatibility
All changes are additive. No endpoints removed. No response field removals.
