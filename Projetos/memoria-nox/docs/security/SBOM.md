# Software Bill of Materials (SBOM) — memoria-nox

> *A complete inventory of all software components in this system, for supply chain transparency and security auditing.*

---

## What is an SBOM?

A Software Bill of Materials (SBOM) is a formal record listing every software component used by a project — direct dependencies, transitive dependencies, licenses, versions, and known vulnerabilities. The US Executive Order 14028 (2021) and CISA guidance established SBOMs as a baseline supply chain transparency requirement for software products.

For memoria-nox, the SBOM:
- Lists all npm packages (direct + transitive) for each sub-package
- Records their SPDX license identifiers
- Maps to known CVEs at generation time
- Is regenerated on every release and available as a GitHub Actions artifact

---

## Format

memoria-nox uses **CycloneDX 1.4** (JSON format), generated by `@cyclonedx/cyclonedx-npm`.

CycloneDX is an OWASP-sponsored standard with first-class npm ecosystem support. It is the format used by GitHub's dependency graph and Dependency Review Action, and is accepted by CISA and NIST NVD.

Alternative: `syft` (Anchore) can also generate CycloneDX or SPDX 2.3 from the same inputs. The CI uses `@cyclonedx/cyclonedx-npm` for accuracy in npm package metadata; `syft` is available for container image SBOMs if memoria-nox ships a Docker image in the future.

## Packages Covered

memoria-nox is structured as several sub-packages. Each gets its own SBOM:

| SBOM file | Covers |
|-----------|--------|
| `sbom/sbom-root.cdx.json` | Root (future main package) |
| `sbom/sbom-staged-a2.cdx.json` | staged-A2 (archive + encryption) |
| `sbom/sbom-staged-a3.cdx.json` | staged-A3 (provider abstraction) |
| `sbom/sbom-staged-p1.cdx.json` | staged-P1 (answer primitive) |
| `sbom/sbom-staged-p5a.cdx.json` | staged-P5a (event bus) |
| `sbom/sbom-staged-l3.cdx.json` | staged-L3 (confidence/provenance) |
| `sbom/sbom-staged-l4.cdx.json` | staged-L4 (typed-link extraction) |
| `sbom/sbom-staged-privacy.cdx.json` | staged-privacy (PII filter) |
| `sbom/sbom-eval-latency.cdx.json` | eval/latency (benchmark harness) |

Generated SBOMs are uploaded as GitHub Actions artifacts on every push to `main` and every tagged release.

## How to Generate Locally

### Prerequisites

```bash
node --version  # must be 20+
npm --version   # 9+ recommended
```

### Install the generator

```bash
npm install --save-dev @cyclonedx/cyclonedx-npm
```

Or install globally:

```bash
npm install -g @cyclonedx/cyclonedx-npm
```

### Generate for a specific package

```bash
# From the repo root
cd staged-A3
npm ci --ignore-scripts
cyclonedx-npm --output-format JSON --output-file ../sbom/sbom-staged-a3.cdx.json
```

### Generate all (convenience script)

```bash
# Once npm run sbom is configured in root package.json:
npm run sbom
```

The `sbom` script is defined in `package.json` and iterates over all sub-packages. It creates `sbom/` at the repo root.

### Script definition (to add to root package.json)

```json
{
  "scripts": {
    "sbom": "node scripts/generate-sbom.js"
  }
}
```

`scripts/generate-sbom.js` iterates over all `staged-*/package.json` and `eval/*/package.json` directories, runs `cyclonedx-npm`, and writes to `sbom/`.

## How to Read the SBOM

Each `*.cdx.json` file follows CycloneDX schema. Key fields:

```json
{
  "bomFormat": "CycloneDX",
  "specVersion": "1.4",
  "metadata": {
    "component": {
      "name": "nox-mem-providers",
      "version": "0.1.0-T1-T8",
      "type": "library",
      "purl": "pkg:npm/nox-mem-providers@0.1.0-T1-T8"
    }
  },
  "components": [
    {
      "name": "typescript",
      "version": "5.4.0",
      "type": "library",
      "licenses": [{ "license": { "id": "Apache-2.0" } }],
      "purl": "pkg:npm/typescript@5.4.0"
    }
  ]
}
```

To scan for vulnerabilities against the SBOM:

```bash
# Using grype (Anchore)
brew install grype   # macOS
grype sbom:./sbom/sbom-staged-a3.cdx.json

# Using osv-scanner (Google)
pip install osv-scanner
osv-scanner --sbom sbom/sbom-staged-a3.cdx.json
```

## Signature Verification (Future)

Once the project reaches its first tagged release (`v1.0.0`), SBOMs will be:

1. **Signed** with Sigstore Cosign (keyless signing via GitHub Actions OIDC)
2. **Attached** to the GitHub release as an artifact
3. **Verifiable** via:

```bash
cosign verify-blob \
  --certificate sbom-staged-a3.cdx.json.cert \
  --signature sbom-staged-a3.cdx.json.sig \
  --certificate-identity-regexp "https://github.com/totobusnello/memoria-nox" \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  sbom/sbom-staged-a3.cdx.json
```

This workflow is defined in `.github/workflows/sbom.yml` (see delivery alongside this doc).

**Current status:** Artifact upload only (no signing). Signing will be activated when the first stable release is tagged.
