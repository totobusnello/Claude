## 3. Highlights técnicos

### 3.1 Content filter crash + manual recovery (PR #57)

O agente de Wave F responsável pelo PR de GitHub hygiene travou ao tentar escrever o texto verbatim do Contributor Covenant 2.1 em `CODE_OF_CONDUCT.md`. O content filter do modelo foi acionado pelo texto de policy de conduta (provavelmente pelo vocabulário de harassment/discrimination — texto corriqueiro em CoC, inofensivo em contexto mas flaggeado fora de contexto).

O worktree do agente travado tinha arquivos parcialmente escritos. A sessão principal:
1. Inventariou o que o worktree tinha (SECURITY.md, CHANGELOG.md, templates `.github/`)
2. Criou `CODE_OF_CONDUCT.md` com referência-por-link ao Contributor Covenant 2.1, sem texto verbatim
3. Comitou + fez push do que estava pronto + abriu PR #57 diretamente

Nenhum trabalho foi perdido. PR #57 está mais robusto do que a versão verbatim seria: a abordagem link-only é preferível porque o texto canônico vive em `https://www.contributor-covenant.org/version/2/1/code_of_conduct/` e não precisa ser duplicado no repo.

### 3.2 THREAT-MODEL recursão: fixing gaps revela novos gaps

O ciclo de Wave E (entrega do THREAT-MODEL.md inicial, PR #55) → Wave F (análise dos módulos pendentes, PR #58) demonstrou que **threat modeling tem recursão estrutural**: analisar novos módulos com rigor invariavelmente revela riscos não cobertos pelo scope original.

PR #58 adicionou §7.5–7.8 (P5 SSE viewer, L2 conflict API, P2 hooks API, A2 orchestrator) e encontrou 7 novos gaps:

| Gap | Severidade | Descrição | Módulo |
|---|---|---|---|
| G11 | **HIGH** | SSE sem limite de conexões concorrentes → DoS por exaustão | P5 viewer |
| G12 | **HIGH** | `/api/hooks/recent` 500 vaza path raw do DB no message de erro | P2 hooks |
| G13 | Med | P5 SSE query exposure via `NOX_VIEWER_SHOW_QUERY` default-off (já mitigado, mas configurável) | P5 |

Positivo: três controles verificados como sólidos sem gaps adicionais — path traversal no viewer (3 camadas independentes), AAD confusion em A2 (GCM auth tag + sha256), append-only triggers em v21 (3 triggers CWE-693).

### 3.3 G4-G10 bundle: 5 gaps, 5 staged dirs, deploy seletivo

PR #59 implementa 5 correções de security em arquitetura `staged-G*/edits/` — o mesmo padrão dos staged-sprint/ dirs de Waves B/C. Cada gap tem seu próprio subdiretório com código, testes e docs, permitindo deploy independente no VPS:

- **G4** — `validateBody()` rejeita `top_k > 20`, `max_tokens > 8192`, `temperature > 1` com 422 estruturado
- **G6** — `requireLocalhost()` com `timingSafeEqual` via sha256; `AUTH-MODEL.md` documenta política completa
- **G7** — `TarBlockTransform` streaming: peak memory O(maior entry) vs O(archive completo) anterior
- **G8** — `protect-audit-db.sh`: `audit.db` separado + `chattr +i` (imutável mesmo como root) + `verify-audit-hardening.sh`
- **G10** — triggers SQL `AFTER INSERT` que overridam `started_at`/`ran_at` backdatados + `validateAuditTimestamp()` 24h drift window

A chave arquitetural: **G8** cria um DB separado para audit e aplica `chattr +i`. Isso significa que mesmo um atacante com root não pode deletar ou truncar o audit log sem usar `chattr -i` explícito (que deixaria trace em logs de kernel). Defense in depth documentado em `docs/AUDIT-HARDENING.md`.

## 4. Lessons learned

### 4.1 Content filter crash: padrão de recovery manual

**O que aconteceu:** agente Wave F travou via content filter ao copiar Contributor Covenant 2.1 verbatim. O texto de policy de conduta (que menciona tipos de harassment, discrimination, etc.) foi flaggeado fora de contexto.

**O que funcionou:**
- Worktree parcial não foi perdido — arquivos que o agente já havia escrito (SECURITY.md, CHANGELOG.md, templates) continuaram disponíveis no path do worktree
- Sessão principal inventariou o worktree, identificou o que faltava (CODE_OF_CONDUCT.md) e completou manualmente
- Abordagem link-only para CoC é arquiteturalmente superior: texto canônico em URL oficial, não duplicado no repo

**Pattern para próximas waves:**
```
Se agente trava via content filter em artefato parcialmente completo:
1. Identificar o worktree do agente travado (path em .claude/worktrees/)
2. Inventariar o que foi escrito antes do crash
3. Identificar o que causa o filter (geralmente: verbatim text de policy, ToS, CoC)
4. Substituir por referência-por-link ou paraphrase que preserva intent sem text verbatim
5. Completar o PR manualmente da sessão principal
6. Documentar no PR body: "PR completed manually after agent content-filter crash"
```

**Nova memória:** `feedback_content_filter_crash_coc_verbatim` — CoC verbatim dispara filter; usar link-only.

### 4.2 Sparse-checkout em worktrees: deve estar no template de spawn

Esta é a **terceira ocorrência** nesta sessão de agente precisando de `git sparse-checkout add <dir>` antes de conseguir escrever em diretório não presente no worktree. Os três casos:

- Wave C: agente precisou de `docs/post-mortems/` (não estava em sparse-checkout padrão)
- Wave E: agente precisou de `docs/` completo para THREAT-MODEL.md
- Wave F (este agente): precisou de `docs/post-mortems/` + `docs/` para escrever post-mortem

**O problema:** o spawn de agentes em worktrees usa sparse-checkout com conjunto mínimo de diretórios. Diretórios de docs adicionais não estão no set default.

**Fix para próximas waves:** adicionar ao template de spawn de agente:
```bash
# Antes do primeiro git add — verificar sparse-checkout
git sparse-checkout add docs/  # ou o dir que o sprint precisa
git sparse-checkout list        # confirmar que aparece
```

Alternativamente: incluir `docs/` no set sparse-checkout default para todos os agentes.

### 4.3 Threat model recursão: sugestão de cadência trimestral

Wave E gerou THREAT-MODEL v1.0 (10 gaps G1–G10). Wave F gerou v1.1 com 7 novos gaps (G11–G17). Isso confirma um padrão: cada onda de módulos novos produz nova superfície de ataque.

**Sugestão arquitetural:** em vez de threat model pontual por wave, estabelecer cadência explícita:
- **Por wave** (durante Wave E/F etc.): seção §7.N para cada módulo novo entregue
- **Trimestral**: revisão de todos os gaps abertos + reclassificação de severidade + sweep de módulos não cobertos
- **Pré-GTM Phase 2**: pentest formal ou security audit externo antes de abrir para terceiros

O template de §7.N no THREAT-MODEL.md já é suficientemente padronizado (STRIDE, controles, mitigações, gaps) para que o processo seja repetível sem overhead alto.

### 4.4 Waves C+D+E+F: padrão de entrega sustentada confirmado

4 waves consecutivas (C, D, E, F) entregaram PRs com:
- Zero BLOCKED.md em qualquer sprint (com exceção do crash de content filter que foi resolvido via recovery manual, não BLOCKED)
- MANDATORY CLOSURE STEPS produzindo URL verificável em todos os agentes que completaram
- Staged-dirs pattern consistente (staged-G*/, staged-sprint/) permitindo deploy seletivo

O único gap: PRs Wave F estão OPEN (não merged) ao tempo do post-mortem. Diferentemente das Waves B, C, D que encerraram com PRs merged antes do post-mortem ser escrito, Wave F ficou com PRs em revisão. Isso é esperado: PRs de security tipicamente aguardam review antes de merge.
