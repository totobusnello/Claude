### 3.3 G16 export-locking previne reutilização de nonce AES-GCM (#66)

Esta é a vulnerabilidade mais severa do bundle, apesar da classificação formal "Medium": em exports concorrentes (dois `POST /api/export` simultâneos), a implementação original não garantia unicidade de nonce por operação. AES-256-GCM com nonce reutilizado não é "um pouco menos seguro" — é **criptograficamente quebrado**: atacante com dois ciphertexts sob o mesmo nonce pode recuperar a chave via XOR.

O fix em `staged-G16/edits/src/lib/export-locking.ts`:
- Mutex de exportação global (uma exportação por vez)
- Nonce derivado de `crypto.randomBytes(12)` + timestamp + PID — collision impossível mesmo em modos de teste
- Teste de regressão `export-concurrent.test.ts`: lança 10 exports simultâneos e verifica que nonces são todos distintos

Classificado "Medium" no THREAT-MODEL porque a janela de exploração exige dois exports concorrentes — improvável em uso pessoal. Em uso multi-usuário futuro, seria imediatamente HIGH.

## 4. Lessons learned

### 4.1 Threat-model recursion: cadência trimestral como resposta

O ciclo observado nesta sessão:
- **Wave E** → THREAT-MODEL v1.0: 10 gaps encontrados (G1–G10)
- **Wave F** (PR #59) → 5 gaps corrigidos (G4, G6, G7, G8, G10)
- **Wave F** (PR #58) → análise de módulos pendentes → **7 novos gaps encontrados** (G11–G17)
- **Wave G** (PR #66) → todos 7 corrigidos
- **Wave F late-merged** (PRs #62, #64) → G1, G5, G2 corrigidos

Isso não é bug no processo — é propriedade estrutural de threat modeling: analisar módulos com rigor invariavelmente expõe riscos que o scope original não cobria. A implicação prática: **não existe "threat model finalizado"**. A estrutura `THREAT-MODEL.md` com seções `🟡 Not shipped` foi a chave — manteve o que estava pendente visível sem bloquear outros trabalhos.

**Cadência recomendada:** auditoria trimestral do THREAT-MODEL com os mesmos módulos de análise (G-N prefix para novos, R-N para recomendações). Cada audit provavelmente encontrará 3-8 gaps novos — isso é saudável, não regressão.

### 4.2 Cross-pillar shims: testes de contrato independentes de refatoração

O padrão de shim usado em PR #65 (testes cross-pillar importam interfaces de pilar, não módulos internos) tem uma propriedade importante: os testes sobrevivem a refatorações internas desde que o contrato da interface seja preservado. Isso é a diferença entre testes de integração frágeis e **contract tests** robustos.

Padrão:
```typescript
// Em vez de:
import { encrypt } from '../../src/lib/archive/encrypt.js';

// Usar shim de pilar:
import { A2Client } from '../shims/a2-shim.js';
const client = new A2Client({ db: ':memory:' });
```

O shim encapsula a inicialização do pilar (schema + migrations + real better-sqlite3). Qualquer test que usa o shim automaticamente valida o contrato de init + a operação testada. Bug-class declarado no arquivo força o autor a articular o risco antes de escrever o teste.

### 4.3 Honesty discipline escala sistematicamente

Aplicada em 4 artefatos de naturezas diferentes nesta sessão:
- **Q4 COMPARISON** (PR #47): `❓` onde harness não rodou
- **README.md** (PR #46): apenas métricas de PRs merged, sem projeções
- **DEMO-VIDEO-SCRIPT.md** (PR #63): pending-Q4 excluídos explicitamente do Hero cut
- **cost-model.md + PRICING-STRATEGY.md** (PR #69): `[H]` em todo valor hipotético; 18 perguntas abertas explícitas

O padrão que funciona: **em cada artefato, declarar explicitamente a política de honestidade** (não apenas praticá-la). Leitores e futuros agentes sabem o que está verificado vs estimado vs hipotético sem precisar reconstituir a história de onde os números vieram.

A consequência: quando os números Q4 chegarem (post Q1+Q2+Q3 runs), a atualização será cirúrgica — apenas substituir `❓` e `[H]` pelos valores reais. Sem risco de "número inventado virou canônico".

### 4.4 Documentation-as-deliverable não bloqueia code waves

Wave H entregou ~3,000 LOC de docs de ops/estratégia (runbooks DR+BACKUP+MONITORING, cost model, pricing strategy, ROI calculator, DOCKER.md) **em paralelo** com as últimas code waves. Docs não bloqueiam em código — podem correr em worktrees separados ao mesmo tempo.

O padrão funciona porque:
1. Docs de ops referenciam APIs e comportamentos **já implementados** (não especulativos)
2. O output é auditável via PR diff como qualquer código
3. Gaps são documentados honestamente (ex: F09 off-site backup rejeitado 2× — documentado como risco informado em ambos DISASTER-RECOVERY.md e BACKUP-RUNBOOK.md, não omitido)

**Wave I e além:** este post-mortem, ROADMAP/HANDOFF/DECISIONS sync, ADRs formais — todos podem correr em paralelo com qualquer wave de código subsequente.

### 4.5 Sparse-checkout: bug recorrente — incluir no spawn template (4ª ocorrência)

Quarta ocorrência nesta sessão de agente precisando `git sparse-checkout add <dir>` antes de conseguir criar arquivo em diretório não presente no worktree. As quatro ocorrências:
1. Wave B: algum sprint tentou escrever em `src/` não presente
2. Wave E: THREAT-MODEL.md em `docs/security/`
3. Wave F: post-mortem em `docs/post-mortems/`
4. Wave G: este post-mortem (mesmo padrão)

**Correção definitiva:** o template de spawn de agente deve incluir:
```bash
# Após criar branch, antes de qualquer write:
git sparse-checkout add docs/post-mortems docs/security docs/ops docs/marketing
git sparse-checkout add src tests scripts sdk
```
Ou simplesmente: `git sparse-checkout set --no-cone '*'` para worktrees de escrita que precisam de acesso amplo.
