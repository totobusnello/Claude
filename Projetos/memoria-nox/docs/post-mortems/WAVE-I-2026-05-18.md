## 4. Lessons learned

### 4.1 Docs live têm drift inevitável — sync micro-PR a cada 2-3 waves

O sync em PR #72 foi necessário porque 6 waves passaram sem atualização dos canônicos. Custo: ~270 linhas de diff, descoberta de 5 itens defasados, risco de futuras sessões operarem com mapa desatualizado.

**Pattern recomendado:** ao final de cada wave (ou no máximo a cada 2–3 waves), um micro-PR de docs-only (ROADMAP + HANDOFF) com ~30–50 linhas corrige o drift antes que acumule. Pode correr em paralelo com o post-mortem da mesma wave — não bloqueia nada.

**Anti-pattern evidente:** deixar docs sync para uma wave dedicada significa que waves intermediárias (G, H) rodaram sem saber exatamente onde o ROADMAP apontava. Nenhum dano real ocorreu nesta sessão, mas o risco de conflito de entendimento entre agentes paralelos cresce com o drift.

### 4.2 ADRs capturam consequences negativas que prosa livre omite

O exercício de converter DECISIONS.md para ADRs revelou uma assimetria consistente: a prosa de DECISIONS.md documenta *o que foi decidido* e *por quê*, mas raramente documenta *o custo*. O template ADR forçado exige seção "Consequences (Negative)", e preencher essa seção sistematicamente revelou trade-offs que haviam ficado implícitos:

- ADR-003: 6.8× latency em import AES-GCM (mensurável, aceitável)
- ADR-005: shadow gating atrasa feature release em ~7d por iteração (aceitável, evita regressão)
- ADR-006: real DB em testes aumenta tempo de setup de ~10ms para ~200ms por suite (aceitável, evita mocks frágeis)

Nenhuma dessas consequences é problemática — mas tê-las documentadas significa que future contributors não vão "descobrir" o trade-off como surpresa ao implementar.

### 4.3 Visual regression vs. snapshot test: Playwright headless + CI trigger cirúrgico

A abordagem de regressão visual de PR #75 usa Playwright headless (não screenshot manual), com trigger de CI configurado para modificações *apenas* nos paths relevantes (`staged-P5/`, `src/viewer/`, `static/`). Isso evita o problema comum onde CI visual roda em todo PR e é lento — aqui roda apenas quando o viewer pode ter mudado.

Os 15 snapshots cobrem os estados de erro e borda (connection limit, auth denied) além dos happy paths. Regressão visual só é útil se captura os estados que regridem silenciosamente — estados de erro são os candidatos mais prováveis.

### 4.4 OpenSSF audit "honest-first": not-met e partial documentados sem euphemismos

O documento `docs/security/OPENSSF-AUDIT.md` de PR #76 lista os 2 not-met e os 15 partials com franqueza igual à usada no Q4 COMPARISON. Nenhum "planned" ou "in progress" para coisas não entregues. Os 2 not-met são explicitamente classificados como "requer ação humana fora do repositório" (submissão web + GitHub App install) — não como bugs ou técnica dívida.

Essa disciplina de honestidade é consistente com o padrão estabelecido em Wave D (❓ markers no COMPARISON.md) e Wave H ([H] markers no cost model). Quando o OpenSSF badge for submetido e Renovate instalado, a atualização será cirúrgica: trocar "Not-met" por "Met" com data de submissão.
