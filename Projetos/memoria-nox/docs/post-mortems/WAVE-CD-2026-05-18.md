## 3. Highlights técnicos

### 3.1 L2 conflict detection — estrutural vs textual (memanto Gap #5)

L2 implementa detecção de contradições no KG via **SQL `GROUP BY` determinístico** sobre relações, não via embedding similarity + NLI como o memanto faz. A diferença arquitetural é significativa: a abordagem memanto detecta contradições em texto livre (semântica); a abordagem nox-mem detecta contradições em relações estruturadas (`kg_relations.source_entity_id / target_entity_id / relation_type`). Type 1 (contradição direta: mesma entidade, mesma relação, valores incompatíveis) é end-to-end no PR #51. Types 2/4 e Type 3 (supersessão temporal) têm hooks no schema v21 mas implementação deferida por spec §11.

Iron laws do sprint: `NOX_CONFLICT_MODE=disabled` default (shadow-first), zero auto-resolução em v1, `conflict_audit` append-only com triggers CWE-693, threshold `min_confidence=0.5`.

### 3.2 L3 confidence field com ranking GATED por eval lift (regra crítica #5)

L3 divide o escopo em dois DoDs distintos: **DoD-A** (write-side hooks + CLI/HTTP/MCP mark workflow + telemetria `/api/health.confidence`) **sempre embarca**; **DoD-B** (integração do confidence score no ranking do search) fica com `NOX_RANKING_CONFIDENCE=disabled` por padrão e só é ativado após eval lift ≥+1.0pp nDCG@10 (n≥200, p<0.05) + 7d shadow + GO explícito. Isso honra a regra crítica #5 do CLAUDE.md: scoring change em commit de "fix" é proibido; ranking é feature work com gate de evidência.

PR #48 tem 99 testes em 12 arquivos de test, incluindo conformance tests que fazem `JSON.stringify` scan verificando que nenhum campo de prompt ou confidence score vaza em erros ou telemetria.

### 3.3 QA matrix 6/6 pacotes green — 511/514 testes

PR #50 rodou `npx tsc --noEmit` + `npm test` em todos os 13 diretórios `staged-*/`. 6 têm `package.json` + `tsconfig.json` e são self-contained; 7 são bundles de patch VPS (SQL, bash, rsync) documentados como "deploy-validated-on-VPS". Resultado: 511/514 testes passando. Os 3 skips em `staged-A3` são chamadas de rede ao Gemini, corretamente gateadas para CI.

**Achado relevante (staged-L3 stale dist):** primeiro run mostrou 2 falhas espúrias (T2.8 + migration ENOENT). Root cause: `dist/` pré-existente de build incompleto anterior. `npm test` dispara `npm run build`, então qualquer cold start é limpo — 99/99 em cold. Nenhum fix de código necessário, achado documentado no PR.

## 4. Lessons learned

### 4.1 Cherry-pick refinado como recovery de worktree leak (refinamento do Wave B 4.1)

Wave B documentou o pattern de worktree branch leak. Wave C+D refinou a recovery: quando commits aterrisam na branch errada, `git cherry-pick <sha>` na branch correta + `git branch -f <wrong-branch> <pre-commit-sha>` para resetar. O insight novo: **não usar merge para "limpar"** — o diff do PR fica confuso e pode incluir mudanças não relacionadas. Cherry-pick preserva o commit exato com sua mensagem original. Pattern agora em memória `feedback_worktree_branch_leak_to_main.md`.

### 4.2 Honestidade em docs estratégicos — marcadores `❓` e `pending`

O padrão estabelecido em Wave D: qualquer métrica sem fonte verificável recebe `❓` explícito; qualquer número de competitor marcado como "vendor-reported, não verificado independentemente". O COMPARISON.md e o COMPETITIVE-POSITIONING.md estabelecem isso como convenção. Implicação prática: na hora de preparar pitch ou comparação pública, sabe-se exatamente o que é verificado vs. o que precisa de rodada de harness.

O README segue a mesma disciplina: Wave B numbers citados com PR source (ex: "p95 ~101.7ms via mock LLM — PR #40"), nunca como fatos absolutos sem contexto.

### 4.3 9 PRs paralelas com zero BLOCKED — MANDATORY CLOSURE STEPS como invariante

Wave C+D manteve 9 PRs paralelas com zero BLOCKED.md, continuando o padrão de Wave B (5/5). O pattern MANDATORY CLOSURE STEPS — com `gh pr view <num> --json url --jq .url` como step explícito — é o que diferencia "PR presumidamente criado" de "PR confirmado existente". Todos os 9 agentes retornaram URL verificável.

Pattern de merge: `--auto` continua preferível para waves com ≥3 PRs simultâneos (GitHub serializa server-side). Sem retry manual de race condition nesta wave.

### 4.4 L2/L3 schema-additive com rollback SQL

Ambos os sprints (L2 e L3) usam o padrão de migrations aditivas: `v21-conflict-audit.sql` + `v21-rollback.sql` (L2) e `v22-confidence-eval-log.sql` + `v22-rollback.sql` (L3). Migrations são idempotentes; rollback SQL incluído no mesmo PR que a migration. Gate via env var (`NOX_CONFLICT_MODE=disabled`, `NOX_RANKING_CONFIDENCE=disabled`) — code presente em main, feature desativada até gate de evidência.

Isso permite que o schema evolua em main sem necessidade de deploy coordenado com feature activation.

### 4.5 Memory consolidation como trabalho paralelo, não serial

As 8 memórias foram escritas como work-item independente (Wave D), em paralelo com L2 e L3. Custo: zero CI, zero conflito, zero bloqueio. Ganho: lições de Wave B preservadas imediatamente enquanto ainda frescas, sem esperar próxima sessão.

Regra geral: qualquer artefato que não tem dependência de CI (docs, memórias, specs, positioning) pode e deve rodar em paralelo com implementações. Não há razão para serializar.

### 4.6 executor-high vs executor — validado em greenfield 300-500 LOC

PR #51 (L2, +4,443 LOC, 30 arquivos novos) e PR #48 (L3, +3,783 LOC, 30 arquivos novos) foram greenfield de módulo complexo com interações semânticas não triviais (conflict audit chain, confidence ranking gate). `executor-high` (Opus) produziu zero false-starts e abstrações limpas. Para T-suffix continuations (extending existing code), docs e test additions: `executor` (Sonnet) suficiente, custo 3-4× menor. Pattern agora em memória `feedback_executor_high_vs_executor_tradeoff.md`.
