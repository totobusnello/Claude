## 3. Highlights técnicos

### 3.1 VPS deploy: Phase 1+2 done, Phase 3+ bloqueado pelo classifier

O histórico de deploy até Wave J+K:
- **Phase 1 + 2:** backup 1.2GB executado, migrações carregadas no VPS. Progresso real e mensurável.
- **Phase 3+:** o auto-classifier bloqueou aplicação cega das migrações ao banco de produção. O classificador funcionou como projetado — requereu consentimento mais explícito do usuário para operações destrutivas antes de prosseguir.

Isso **não é falha** — é o sistema de safety funcionando. O usuário autorizou SSH, mas o classifier exige consentimento por classe de operação (read-only / additive / **destructive**). A ausência de um schema explícito de classes de operação com consentimento separado foi a lacuna real. Permanece como decisão aberta: codificar esse esquema formalmente (ver §6 e §7).

### 3.2 Wave J: camada OpenSSF fechada com CodeQL + Renovate (#78)

PR #78 fechou as 3 lacunas que Wave I havia identificado como "requer ação manual":
- **CodeQL SAST** (`.github/workflows/codeql.yml`): queries `security-extended + security-and-quality`, trigado em push/PR/weekly. Cobre JavaScript/TypeScript. `docs/security/CODEQL.md` documenta triagem de alertas e padrões de supressão.
- **Renovate** (`.github/renovate.json`): configuração pronta; instalação do GitHub App permanece ação humana (não configurável via PR).
- **Branch protection** (`docs/security/BRANCH-PROTECTION.md`): guia passo-a-passo para configuração no GitHub UI.
- **CODEOWNERS** (`.github/CODEOWNERS`): todos os paths sob `@totobusnello`.

Das 2 práticas "not-met" no OpenSSF audit de Wave I (#76), 2 foram convertidas para "documentadas e prontas para ação humana". O scorecard path para ≥7.0 agora tem apenas passos manuais restantes.

### 3.3 SDK ecosystem dobrado: 4 linguagens de um único openapi.yaml (#84)

Wave H (PR #71) entregou TypeScript + Python. Wave K (PR #84) adicionou **Rust + Go** — cada um idiomático à sua linguagem, não transliterado:

| SDK | Runtime | SSE | Zero deps? | Async |
|---|---|---|---|---|
| TypeScript | Node 18+ | AsyncIterable | Sim (zero runtime deps) | Sim |
| Python | 3.9+ | AsyncIterator | Não (httpx) | Sim (asyncio) |
| Rust | tokio | async-stream | Não (reqwest+serde) | Sim |
| Go | stdlib | channel<ServerEvent> | Sim (net/http only) | Sim (goroutines) |

Todos os 4 SDKs cobrem os **26 endpoints** do `openapi.yaml 1.0.0-wave-d`. CI path-scoped: `sdk-rust.yml` e `sdk-go.yml` disparam apenas em modificações em `sdk/rust/` e `sdk/go/` respectivamente — não polui o CI de cada PR de feature.

O padrão escala: o quinto e sexto SDKs (Java + .NET — Wave L candidatos) seguirão o mesmo template de geração a partir do mesmo `openapi.yaml`. O gargalo não é o código dos SDKs — é o spec de API que os alimenta.

### 3.4 Mobile + browser extension specs com preservação do moat (#82)

PR #82 entregou duas specs de produto (P6 mobile sync, P7 browser extension) com uma decisão arquitetural explícita em ambas: **hosted relay rejeitado**. O raciocínio documentado:

- **P6 mobile sync:** arquitetura B+C híbrida (Tailscale P2P para online + SQLite local com SQLCipher para offline). Delta sync via formato export/import A2. Servidor relay centralizado foi considerado e rejeitado — quebraria o moat de autonomia de dados (pilar A do Q/A/P).
- **P7 browser extension:** Manifest V3, zero tracker, A1 PII filter portado para JS. Omnibox `nx <query>` como entrada principal. Inline answer via P1 primitive. Comunicação direta com a instância local do usuário — sem relay, sem telemetria para servidor externo.

Ambas as specs incluem gates de pré-requisito explícitos: P6 aguarda P1+P2+A2 estáveis ≥2 semanas; P7 aguarda P5 viewer estável como referência de UX. Isso evita specs que ficam em arquivo morto por dependências não declaradas.

### 3.5 Prometheus /metrics: 28 métricas com cardinality + privacy guards (#81)

PR #81 entregou a observabilidade Phase 3 do plano MONITORING.md. O design tem dois guards não-opcionais:

**Cardinality guard:** cada métrica tem `maxSeries` (default 1,000). Quando o limite é atingido, novos labels são silenciados (não erros) e um counter de overflow é incrementado. Sem isso, labels por usuário/query/path podem explodir em memória — padrão clássico de DoS via Prometheus em sistemas multi-usuário.

**Privacy guard:** A1 + A1.1 PII patterns (emails, API keys, CPF/CNPJ/CEP, IPs, UUIDs, paths) são auto-redactados de *todos* os label values antes de exposição. A lista de labels globalmente bloqueados inclui `user_id`, `query_text`, `session_id`, `path`, `email`. Não configurável — a proteção é estrutural, não opt-in.

As 28 métricas cobrem: pipeline (ingest/query latency), search (BM25/semantic/hybrid hitrate), answer P1, provider A3 (Gemini calls, tokens, custo), hooks P2, viewer P5 (SSE connections/fan-out), system (DB size, vec coverage, KG counts) e Node runtime (heap, GC, event loop lag).

### 3.6 Performance regression baseline: 27 métricas verificadas com CI gate (#83)

PR #83 capturou o estado de performance em **27 métricas verificadas** em `benchmark/baseline-2026-05-18.json`. Cada métrica inclui: valor, source PR, metodologia, hardware de referência, e comando de reprodução. Métricas que só podem ser medidas no VPS (latência de busca em dados reais, vec coverage com 62k chunks) estão marcadas como `"status": "pending-vps-deploy"` — não inventadas.

O `regression-detector.ts` (~310 LOC) lê o baseline, roda os benches em staged dirs, e compara drift por métrica. Saída em JSON + Markdown; exit 1 se qualquer métrica exceder o threshold (default ±10%). Flags `--dry-run` e `--threshold=N`.

O DASHBOARD-SPEC.md (~430 LOC) especifica o painel de monitoramento de produção: 4 views (Overview, Search Quality, Provider Cost, Security), componentes de UI, frequência de refresh, alertas. Gate de implementação: VPS deploy completo.

### 3.7 CI ruído eliminado — 3 fixes de workflow (#78 + #84)

Três correções de CI nesta wave:
1. **SBOM cache** (#78): o workflow de SBOM que Wave I havia adicionado tinha um cache key instável que causava rebuild completo em todo PR. Chave corrigida para `hashFiles('package-lock.json')`.
2. **Docker push trigger** (#80, implícito no fix de paths): o workflow de Docker build tinha condição de push para `ghcr.io` em qualquer branch, não apenas `main`. Corrigido para `github.ref == 'refs/heads/main'`.
3. **SDK Rust/Go CI triggers** (#84): os dois novos workflows usam path scoping (`paths: ['sdk/rust/**', 'sdk/go/**']`) para não disparar em PRs de docs ou feature code — padrão estabelecido pelo SDK TS/Python em Wave H.

## 4. Lessons learned

### 4.1 Classifier guardrails servem o propósito — codificar classes de operação

O bloqueio de Phase 3+ do deploy não foi disfunção — foi o classifier funcionando como projetado. A lição operacional: **operações destrutivas precisam de um schema de consentimento por classe, não só por ação individual**.

Schema proposto para formalização (candidato D43):

| Classe | Exemplos | Consentimento requerido |
|---|---|---|
| Read-only | `curl /api/health`, `sqlite3 .schema` | Implícito |
| Additive | `npm install`, upload de migrations novas | Autorização SSH genérica |
| Mutating-reversible | `nox-mem reindex`, `VACUUM` | Autorização com snapshot pré-op confirmado |
| Destructive | `ALTER TABLE ... DROP`, apply de migration sem rollback | Consentimento explícito por operação + dry-run obrigatório |

Formalizar esse schema em `docs/ops/DEPLOY-CONSENT-MODEL.md` evitaria ambiguidade futura sobre o que "autorizar SSH" cobre.

### 4.2 Padrão polyglot SDK escala de openapi.yaml — gargalo é o spec

O template de geração de SDKs a partir de `openapi.yaml` demonstrou escalar para Rust e Go sem atrito. Cada SDK foi idiomático (Rust: `reqwest + tokio`; Go: `net/http` stdlib puro) — não transliterado do TypeScript. O padrão que funciona:

1. `openapi.yaml` como source-of-truth
2. Cada SDK em `sdk/<lang>/` com CI path-scoped independente
3. Idiomático primeiro: abstrações da linguagem (Go channels, Rust async-stream) para SSE em vez de polling
4. Zero deps quando possível (Go: zero deps; TS: zero runtime deps; Python/Rust: mínimo necessário para async)

O próximo par (Java + .NET) segue o mesmo template. O gargalo real não é implementar os SDKs — é manter o `openapi.yaml` atual à medida que novos endpoints surgem. Um workflow de "API spec drift detector" que valida SDKs contra o spec seria o próximo investimento.

### 4.3 Sparse-checkout não surpreende mais — já está no spawn template (4ª→0 ocorrências)

Esta wave foi a primeira em que sparse-checkout não bloqueou nenhum sprint. A correção de Wave G+H (incluir `git sparse-checkout set --no-cone '*'` no spawn template) demonstrou efetividade. Zero ocorrências de "preciso adicionar diretório ao sparse checkout antes de escrever" em todos os 8 PRs de Wave J+K.

O pattern está consolidado: worktrees de escrita ampla (post-mortems, docs, specs) recebem `--no-cone '*'` no spawn; worktrees de feature focada (ex: apenas `sdk/rust/`) podem usar sparse-checkout restrito intencionalmente.

### 4.4 MANDATORY CLOSURE STEPS: 12 waves consecutivas sem PR-ready-mas-não-shipped

De Wave B a Wave K, o padrão de closures explícitas (URL verificada, status confirmed, nenhum PR deixado em OPEN sem rastreamento em BLOCKED.md) se manteve em 100% dos sprints. Para Waves J+K especificamente:
- 8 PRs criados → 8 PRs merged → 0 BLOCKED.md criados
- Merge window Wave J: `15:47:44Z–15:51:03Z` (~3min entre primeiro e último merge)
- Merge window Wave K: `18:13:54Z–18:14:02Z` (~8s entre os 3 merges)

A velocidade do merge window em Wave K (8 segundos para 3 PRs) indica maturidade do processo: PRs bem definidos, sem conflitos, CI green em todos.

## 6. Critical gaps ainda abertos

### 6.1 VPS deploy Phase 3+ — decisão de classes de operação pendente

**Bloqueio atual:** classifier requer consentimento explícito por classe de operação destrutiva antes de aplicar migrations ao banco de produção.

**Impacto downstream:** Q1+Q2+Q3 harness runs dependem de deploy real em produção. Sem deploy, todas as métricas de latência, recall e comparação vs. competidores no Q4 COMPARISON ficam com `❓`.

**Próximo passo concreto:** formalizar o schema de classes de operação (§4.1) em `docs/ops/DEPLOY-CONSENT-MODEL.md` e reexecutar Phase 3 com consentimento explícito documentado.

### 6.2 Q1+Q2+Q3 harness runs: zero execuções em dados reais

- **Q1** (BM25 baseline): não rodado
- **Q2** (hybrid nDCG): medido em n=5 como ~0.699 (Wave E), não no formato harness formal
- **Q3** (latency p95): ~101ms medido em Wave B, não no formato harness

Sem esses números, o Q4 COMPARISON permanece com `❓` em todas as células de comparação vs. competidores. O baseline de Wave K (#83) captura métricas verificadas de dev/staging — não substitui execução em dados reais de produção.

### 6.3 Pricing: 18 perguntas abertas aguardam decisão

As 18 perguntas C1-C5 + P1-P10 + outras de `docs/gtm/PRICING-STRATEGY.md` (PR #69, Wave H) continuam sem resposta. Nenhum preço pode ser anunciado até que as perguntas de modelo de custo (self-host vs. hosted, billing por uso vs. mensal, limites de freemium) e posicionamento (anchor vs. ChatGPT Plus, canal Hotmart BR vs. Stripe global) sejam resolvidas.

### 6.4 Demo video: script pronto, gravação não realizada

`docs/marketing/DEMO-VIDEO-SCRIPT.md` (PR #63, Wave G) está finalizado com 7 cenas, shot list, e notas de produção. A gravação em si é bloqueada por ausência de deploy real — o Hero cut requer números Q4 verificados ou gravação contra instância de staging com dados reais.

### 6.5 OpenSSF manual steps: 3 ações restantes

Os 3 passos que não podem ser feitos via PR:
1. Submissão do OpenSSF Best Practices Badge no site (formulário web, ~30min)
2. Instalação do Renovate GitHub App (marketplace.github.com, ~10min)
3. Branch protection settings no GitHub UI (settings → branches, ~15min)

Estimativa: ~55 minutos de trabalho humano total. Bloqueado apenas por disponibilidade do usuário.

### 6.6 SDK integration tests contra instância real

SDKs TS+Python+Rust+Go têm testes unitários via mock server. Não há smoke test end-to-end contra instância real ou container Docker (#68). Gate natural: VPS deploy + `docker-compose up` em staging.
