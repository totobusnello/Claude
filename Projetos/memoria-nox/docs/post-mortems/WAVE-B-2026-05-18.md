## 3. Highlights técnicos

### 3.1 AAD chain bug pego via integration test (A2)

O primeiro round-trip test em A2 falhou porque o orchestrator estava reconstituindo o manifesto de exportação com um `created_at` atualizado, quebando a AAD (Additional Authenticated Data) chain do AES-256-GCM. O campo `created_at` deve ser congelado no manifesto original — qualquer rebuild precisa copiar o timestamp original, não gerar novo. Descoberto via `TamperedArchiveError` no teste de decrypt, não em review manual. Lição: integration tests sem mock do DB/crypto pegam bugs semânticos que testes unitários não pegam.

### 3.2 Latency 42× under budget (P1)

P1 answer primitive mediu p95=101ms contra budget especificado de 4.3s. A decomposição por fase (retrieval / prompt build / LLM / citation parse / telemetry write) mostrou que o gargalo esperado era LLM, mas com flash-lite o round-trip fica em ~80ms. Budget 4.3s foi conservador baseado em flash (não lite). Implicação: o budget pode ser apertado pra ~500ms em produção sem risco.

### 3.3 Bundle 11.7KB vanilla JS (P5)

P5 viewer atingiu 11.7KB total (HTML+JS+CSS) sem bundler, sem React/Vue. Target era <50KB (4× de margem). A chave foi o constraint de design: nenhum `import` no frontend, SSE nativo do browser, CSS `prefers-color-scheme` media query em vez de toggle JS. Dark mode, multi-panel, fan-out, Last-Event-ID resume — tudo em vanilla. Maintainability > bundle size como constraint primário.

### 3.4 80% Gemini calls saved (L4)

L4 production wire implementou confidence gate: wikilinks/md-links com score ≥0.90 pulam LLM completamente, frontmatter typed-links ≥0.95 também pulam. Apenas bare_refs (score 0.75) fazem fallthrough para Gemini. No synthetic corpus de 20 amostras, isso eliminou 80% das chamadas Gemini mantendo 95.8% de precision/recall. O gate é desabilitável via `NOX_L4_SKIP_GEMINI=0` para auditoria.

### 3.5 Cost cap com zero prompt content (A3)

`CostCapExceededError` de A3 expõe apenas contagens (tokens, requests, USD) — nunca prompt content. `redactSecrets` aplicado em `error_kind`. Telemetry write-behind queue bounded em 500 rows (evita OOM em burst). Conformance tests verificam via `JSON.stringify` scan que nenhum campo de prompt vaza em erros ou telemetria.

## 4. Lessons learned

### 4.1 Worktree leak detection

**O que aconteceu:** 3 agentes em rodadas anteriores (wave-a sprints) fizeram push para branches `wave-b/*` a partir de worktrees que haviam "vazado" de volta para `main` como HEAD. Resultado: PR apontava para base `main` mas diff incluía mudanças não intencionais de outros sprints.

**Como detectar:**
```bash
# Antes de qualquer commit em worktree, verificar:
git -C <worktree-path> log --oneline origin/main..HEAD
# Deve mostrar SOMENTE os commits do sprint corrente.
# Se aparecerem commits de outros sprints → worktree contaminado.

git -C <worktree-path> branch --show-current
# Deve ser a branch do sprint. Se for 'main' → bug no spawn.
```

**Recovery:** `git -C <worktree-path> reset --hard origin/main` + recriar branch do sprint + reaplicar apenas as mudanças do sprint (via `git cherry-pick` ou re-implementação). Não tentar "limpar" com merge — o diff fica confuso no PR.

**Prevenção:** no spawn de cada agente, injetar verificação explícita: `git branch --show-current` deve corresponder ao nome do sprint antes do primeiro `git add`.

### 4.2 YAML block scalar dedent bug (CI workflow)

**O que aconteceu:** `.github/workflows/zero-vendor.yml` usava bash heredoc inline no campo `run:` do GitHub Actions:
```yaml
run: |
  BODY=$(cat <<'EOF'
  linha 1
  linha 2
  EOF
  )
```
O parser YAML do GitHub Actions interpreta a indentação do heredoc marker `EOF` como parte do bloco e quebra em parse error silencioso (job simplesmente não roda).

**Fix aplicado (816c3d3):** substituir o heredoc por group de comandos encadeados:
```yaml
run: |
  { echo "linha 1"; echo "linha 2"; } > /tmp/output.txt
```
Ou, para strings multi-linha longas, usar arquivo temporário escrito com `printf '%s\n'`.

**Regra geral:** em campos `run:` de GitHub Actions, nunca usar `<<'EOF'` heredoc. Use `printf`, `echo`, ou escreva em arquivo separado e faça `cat`.

### 4.3 Parallel merge race condition no GitHub

**O que aconteceu:** 5 agents tentaram `gh pr merge --merge <num>` simultaneamente. GitHub retornou `"Base branch was modified"` em 3-4 deles porque o merge do primeiro PR atualizou `main` antes dos outros completarem.

**Comportamento observado:** os merges não falharam silenciosamente — retornaram exit code não-zero com mensagem clara. Agentes com MANDATORY CLOSURE STEPS implementados detectaram o erro e re-tentaram sequencialmente.

**Pattern recomendado para próximas waves:**
```bash
# Opção A — merge sequencial (orquestrador faz merges um a um):
for pr in 38 39 40 41 42; do
  gh pr merge "$pr" --merge --delete-branch
  sleep 2  # GitHub indexa antes do próximo
done

# Opção B — usar --auto (merge automático quando CI verde):
gh pr merge "$pr" --merge --auto
# GitHub serializa automaticamente quando a fila processa.
```
Opção B é preferível para waves futuras: zero coordenação entre agentes, GitHub serializa.

### 4.4 MANDATORY CLOSURE STEPS foi efetivo

Todos os 5 agentes que completaram entregaram PR URL verificável como última linha. Contraste com o incidente `writer-agent-loss` de 2026-05-18 manhã (D41) onde um agente finalizou sem confirmar URL do PR, causando perda de rastreabilidade.

**Pattern que funcionou:**
```
MANDATORY CLOSURE STEPS:
1. git add <files>
2. git commit -m "..."
3. git push -u origin <branch>
4. gh pr create --title "..." --body "..."
5. gh pr view <num> --json url --jq .url   ← retornar isso
```
A etapa 5 forçando retorno da URL é o que diferencia "PR criado" de "PR confirmado existente".
